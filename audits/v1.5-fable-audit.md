# v1.5 — Fable-led end-to-end audit (UI/UX, code, daily operations, PII)

Requested: a full audit covering UI/UX, code, and whether the daily-refresh pipeline actually
works in practice — plus a full current-and-historical PII sweep of the repository. Dimension
reviews were run on a cheaper (non-Fable) model for token efficiency, then independently
fact-checked line-by-line before acting on anything; Fable synthesized the final verdict as
project director. Live GitHub Actions dispatches and a full git-history scan were performed
directly (not simulated), not merely read as code.

## Method

1. **UI/UX dimension review** — cheaper model, read-only review of `index.html`/`styles.css`/
   `app.js`/`figure.js`/`manifest.webmanifest` against BUILD-PLAN.md's design invariants.
2. **Code quality/correctness dimension review** — cheaper model, read-only review of
   `app.js`/`lib.mjs`/`figure.js`/`sw.js`/`generate-daily.mjs`/`verify.mjs`.
3. **Independent fact-check** — every finding from both reviews was re-verified directly
   (precise WCAG contrast math recomputed in Node, claimed missing CSS rules re-grepped, the
   claimed app.js bug traced through the actual control flow) before any fix was applied.
   Two of the UI/UX review's contrast-ratio claims and one "missing cursor:pointer" claim did
   not survive this check and were discarded — a smaller model's hand-computed contrast math
   was wrong; the project's own numeric `verify.mjs` check was right.
4. **Daily-operations live verification** — not a code read. Actually dispatched the
   `daily-cards` and `watchdog` GitHub Actions workflows against the real repo, twice each,
   and inspected the real resulting commits, run history, and live deployed site.
5. **PII / history scan** — `git rev-list --all` + `git grep` across every commit reachable
   from any branch/ref, plus a check of every commit's author/committer identity.
6. **Fable synthesis** — project-director verdict over all of the above.

## Findings and outcomes

### Confirmed and fixed

- **`app.js`**: `renderValues()`/`renderToday()` were called outside the `try`/`catch` that
  wraps the data fetch in `boot()`. A fetch *failure* was already handled correctly (traced:
  `staleness(null)` → `"offline"` → safe local-rotation fallback, no bug there), but if
  `cards.json`/`values.json` ever came back with an unexpected *shape*, the render calls could
  throw uncaught and the page would silently show nothing instead of the intended error card.
  Low probability (these files are schema-validated by `verify.mjs` before every deploy) but a
  real gap against the app's own graceful-degradation design. Fixed by moving both calls inside
  the existing `try` block.
- **`styles.css`**: no `:focus`/`:focus-visible` rule existed anywhere — keyboard users got no
  visible focus indicator on the theme toggle, tabs, or the Fresh card's link (invariant 7 /
  WCAG 2.4.7). Added `--accent`-colored `:focus-visible` outlines on all three.
- **`styles.css`**: `.card`'s `box-shadow` color was hardcoded to the blossom theme's ink RGB,
  so the calm theme's cards cast a faint pink-tinted shadow. Added a `--shadow` custom property
  per theme.
- **Header/figure spacing**: nudged header top-padding 6px → 10px for a touch more breathing
  room below the figure now that it sits at the true page top (v1.4). Minor, not a hard bug.

### Discarded after independent verification (did not survive fact-checking)

- Three claimed WCAG contrast failures (calm `--muted`/`--bg`, blossom `--muted`/`--bg`,
  blossom `--accent`/`--bg`). Recomputed relative luminance by hand in Node for all three:
  **5.05:1, 4.79:1, 4.61:1** — all pass the 4.5:1 requirement. The reviewing model's manual
  contrast math was wrong; `verify.mjs`'s existing numeric check was correct all along.
- Claimed missing `cursor: pointer` on `.tab` — already present in the CSS.

### The headline finding — daily-refresh pipeline was silently broken for future runs

Dispatched `daily.yml` twice, live, back to back, against the unchanged content pool:
**anchor/shift/fresh picks were byte-identical between the two runs** (only the generation
timestamp differed) — genuine same-day idempotency, confirmed in production, not just in a
local dev test. `state/fresh-history.json` also stayed clean (one entry for today, no
duplicates, across three total runs today).

But: **the bot's commits were never triggering the separate Pages-deploy workflow.**
`daily.yml`'s commit-and-push step authenticates with the default `GITHUB_TOKEN`, and GitHub
does not fire other workflows' `on: push` triggers for commits made with that token (documented
loop-prevention behavior). Proved this empirically — two real bot commits landed on `main`
during this audit, and `pages-deploy.yml`'s run history shows **zero** runs triggered by either
one. Net effect, if left unfixed: every real day going forward, the cron job would faithfully
update `data/daily.json` in git, but the *live* site would never actually show it — it would
stay frozen on whatever a human's last real push had deployed. This is exactly the failure mode
the whole feature exists to prevent, and it would have shipped completely undetected.

Compounding it: the existing `watchdog.yml` staleness check only compared the calendar-date
string (`dateHKT`) between the live site and the committed file. It would **not** have caught
the bug above, because within the same calendar day the date string doesn't change even though
the live content is stuck on an earlier same-day generation.

**Fixes applied** (mechanically verified — YAML parses clean, `verify.mjs all` stays 59/59 —
but genuinely **not yet live-tested against a merged `main`**, since testing the fix requires
the fix to be deployed first; see "what's still open" below):

1. `daily.yml` now explicitly dispatches `pages-deploy.yml` via `gh workflow run` immediately
   after a successful commit (using an API call, which is not subject to the same
   loop-prevention restriction as a push event), gated on the commit step actually having made
   a commit. Added `permissions: actions: write` to allow this. Confirmed `pages-deploy.yml`
   already declares `on: workflow_dispatch`, so no change was needed on that side.
2. `watchdog.yml` now also compares `generatedAtISO` (not just the date) between the live site
   and the committed file, so a future same-day stuck-deploy would actually be caught. This is
   safe from false alarms because `watchdog` runs at 09:00 HKT, roughly 3 hours after the
   expected 05:56 HKT refresh — well past any normal CDN cache lag (observed `cache-control:
   max-age=600` on Pages responses), unlike a check run within minutes of a deploy would be.

### Other checks performed, no issue found

- Cron schedules: `daily.yml`'s `56 21 * * *` (UTC) = 05:56 HKT next day; `watchdog.yml`'s
  `0 1 * * *` (UTC) = 09:00 HKT same day — both land correctly per their own documented intent,
  verified by hand (UTC+8 arithmetic), not just by trusting the comments.
  Both workflows' own live dispatches (`pages-deploy.yml`, `watchdog.yml`) completed
  successfully today, independent of the `daily.yml` fix above.
- Fresh-feed-down/empty path: traced `generate-daily.mjs` — if no source produces a valid
  candidate, `fresh` is written as `null` (an explicitly supported outcome per the code's own
  comment); `app.js`'s `renderToday()` correctly treats a null/invalid fresh as "not ok" and
  falls back to the deterministic reserve card. No bug.
- Invariant 2 (zero verbatim-quote glyphs): already enforced by `verify.mjs`'s stage3 check,
  confirmed still passing.
- Animation pacing (26s hold-then-transition sun-salutation loop, v1.4): confirmed it still
  reads as clearly alive within a few seconds; reduced-motion fallback renders a static frame
  correctly.
- Event-listener lifecycle in `figure.js`'s custom element, the animation's segment-lookup
  math, the RSS same-day idempotency fix from an earlier round, and `sw.js`'s network-first
  fetch strategy — all reviewed and confirmed correct.
- ARIA roles/labels on tabs and the theme toggle, Fresh-card `textContent`-only rendering with
  https-only URL validation, no flash-of-wrong-theme — all confirmed correct.

### PII / full-history scan — clean

Searched every commit reachable from any branch (`git rev-list --all` + `git grep`) for: the
one real name relevant here, the site owner's known email, generic email-address and
phone-number patterns, and PII-adjacent phrases ("my wife", "date of birth", street-address
patterns, etc.), plus every commit's author/committer identity and every filename ever
committed across all of history.

- All 20+ commit authors/committers are pseudonymous/service accounts (a GitHub-noreply
  address, an AI-assistant-noreply address, and the `github-actions[bot]` address) — no real
  name or personal email anywhere in git metadata.
- Zero email addresses of any kind in any commit's content, ever.
- The only "phone-number-like" regex hits were the well-known xmur3/mulberry32 PRNG hash
  constants used by the rotation algorithm — confirmed false positives by inspecting context.
- The only "PII-adjacent phrase" hits were the project's own written PII *policy* text
  ("No real name, employer, financial figure, date of birth, or location...") appearing in
  every historical revision of the build plan — describing the rule, not violating it.
- No uploaded/local file paths or unexpected files were ever committed; the full list of every
  filename ever added across history is exactly the expected project file set.

**Conclusion: no PII linking to the site owner or his wife anywhere in current content or full
git history.**

## What's still open after this PR merges

The Pages-redeploy self-dispatch fix is logically sound and passes every check available
pre-merge, but it genuinely cannot be live-tested until it's on `main` (dispatching a workflow
runs the version of the file already committed to the target ref). **Recommend dispatching
`daily.yml` once, shortly after merge, and confirming `pages-deploy.yml` shows a new run
triggered by the resulting bot commit** — if it doesn't, this reopens as a live blocker.
