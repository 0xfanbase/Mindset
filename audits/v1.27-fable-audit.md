# v1.27 — Fable project-director + auditor review (daily refresh boundary 06:00 → 05:00 HKT)

Requested by the owner explicitly: an independent project-director pass to review the plan/diff
before merge, then a separate independent auditor pass to sign off on the final state — the same
two-role convention this repo has used before (v1.1's director review, v1.5/v1.17/v1.20/v1.21's
audits), run back-to-back for one change this time rather than across a longer build.

Scope: the follow-up to the yesterday's-cards investigation logged in `decisions.md` — moving the
daily-refresh boundary from 06:00 to 05:00 HKT. Two commits on
`claude/yesterday-cards-display-bug-f0374t`: `7032ed4` (the original 9-file change) and `8f40fef`
(the two director-requested fixes on top). Both passes had full repo read access and, for the
auditor, `verify.mjs`/`git` execution; neither pass was shown the other's raw transcript, only a
written summary of what came before.

## Method

**Director pass (`Plan` role, read-only):** read CLAUDE.md and BUILD-PLAN.md in full; read the
actual diff via `git show`/`git diff` rather than trusting the implementer's summary;
independently re-derived the UTC/HKT arithmetic for both the cron change and the
`expectedDateHKT` threshold; grepped the whole repo for any missed "06:00"/"05:56"; ran
`verify.mjs all` itself; checked CLAUDE.md invariants 10 and 12 specifically; judged the
watchdog-schedule and missing-changelog decisions on their merits rather than assuming they were
oversights.

**Auditor pass (`general-purpose` role, full tool access), run after the director's required
fixes were applied:** re-read CLAUDE.md/BUILD-PLAN.md; treated the working tree (not just the
last commit) as the state to verify; re-checked both director-requested fixes landed correctly
and didn't overshoot; ran its own full repo sweep for stale time references; re-ran `verify.mjs
all` independently; re-derived the HKT/UTC arithmetic a second time from scratch; checked
invariant 10 against the reflog; and went beyond the brief to check `generate-daily.mjs`, budget
compliance, and whether the new verify.mjs test was actually wired into the stage1 runner rather
than dead code.

## Findings

### Director pass: two must-fix items, both real

- **`watchdog.yml` had a second, separate stale-time reference the first cut missed.** The YAML
  schedule comment ("~3h buffer... 05:56/06:00 HKT daily") had been updated, but a second,
  textually unrelated comment further down inside the inline bash script for the actual
  staleness check ("this runs (09:00 HKT, ~3h after the expected refresh)") had not — easy to
  miss since it isn't adjacent to the schedule block and phrases the same fact differently.
  Fixed: now reads "~4h after the expected refresh", consistent with the rest of the file.
- **An edit had accidentally touched historical changelog narrative.** The v1.16 changelog
  entry's own description of what it shipped ("counts as the focus window (including the
  pre-06:00 hours when yesterday's cards are still showing)") got swept up in a repo-wide
  find-and-replace and changed to "pre-05:00" — factually wrong, since the boundary really was
  06:00 when v1.16 shipped, and editing it would have quietly rewritten history. Reverted,
  matching the same principle this file's own changelog and every `audits/` file already follow
  (superseded history stays as it was, per the v1.20 precedent).

Both were confirmed independently correct by the director (arithmetic re-derived, the
cron+threshold coupling reasoning traced through `staleness()`/`expectedDateHKT()`) before
requiring the fixes; verdict was **CHANGES REQUESTED**, narrowly, with the underlying engineering
already judged sound.

### Auditor pass: both fixes verified correct, full independent re-check, signed off

- Confirmed `watchdog.yml` now has zero remaining "~3h"/"05:56"/"06:00" anywhere in the file,
  both comments consistent at "~4h"/"04:56/05:00 HKT".
- Confirmed the historical-line revert landed exactly right and did not overshoot: all 10
  non-historical BUILD-PLAN.md references still correctly say 05:00/04:56.
- Independently re-swept the whole repo for "06:00", "05:56", "~3h" (plus "6am", "21:56", ">= 6"
  as additional angles) — every hit is preserved history in `audits/` files or the one
  intentional v1.16 line; zero stale live references.
- Re-ran `verify.mjs all` from scratch: 74/74. Confirmed the new boundary test actually appears
  in the stage1 run output (not dead code) and its instants are correct.
- Re-derived the cron/threshold arithmetic a second time, independently: self-consistent with
  each other and the 05:00 target.
- Checked `git reflog`: commit → `pull --rebase origin main` over the bot's daily-cards commit
  → push — exactly the pattern CLAUDE.md invariant 10 prescribes; no amend/rewrite.
- Beyond the brief: `generate-daily.mjs` correctly needs no change (it only stamps
  `hktDateString(now)`; the boundary logic is entirely client-side). No other file references

### False alarms, reported rather than dropped (per this repo's convention)

- **Two same-day `[bot] daily cards 2026-07-23` commits on `main`** initially looked like a

  original stale-site investigation, the other the scheduled cron firing ~49 minutes late per

  so only `generatedAtISO` differed between them.
- **BUILD-PLAN.md's §8.2 watchdog YAML copy has a pre-existing cosmetic mismatch against the

## Verdict

**Signed off — ready to merge.** Both director-required fixes are correct and independently
re-verified; the auditor's own from-scratch re-derivation of the core arithmetic, repo-wide
sweep, and `verify.mjs` run all confirm the shipped state is complete and consistent. Outstanding
before this write-up (now closed alongside it): this changelog entry itself and BUILD-PLAN.md's
title-header bump (pre-existing v1.21 drift, unrelated to this change but closed while here).
One operational note for whoever merges: landing the merge between 04:56 and 05:56 HKT on any
given day would skip that day's scheduled cron run outright (new slot already passed, old slot
gone) — merge outside that window, or dispatch `daily-cards` manually right after merging.
