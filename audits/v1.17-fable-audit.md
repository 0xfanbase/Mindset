# v1.17 — Fable-led UI/UX audit (Kenya trip countdown pill)

Requested by the owner alongside the feature itself: a day countdown on the Kenya card, top
right, for the 2026-08-15 Kenya flight, with a "beautiful UI design" and a Fable audit before
the PR merges. Scope of this audit: the v1.17 diff on branch `claude/kenya-trip-countdown-s4ybik`
(touches `lib.mjs`, `app.js`, `styles.css`, `scripts/verify.mjs`, `BUILD-PLAN.md`). Per this
project's audit convention (v1.5, v1.16), every implementer claim was re-verified independently
— contrast math recomputed from raw token values in Node, fresh screenshots taken at a real
mobile viewport with a frozen browser clock, verifier counts re-run at both the feature commit
and the pre-feature baseline — nothing was accepted on the implementer's word.

Note on repo state: the feature landed as commit `5914445` on the branch while this audit was
in progress. Confirmed `git diff HEAD` is empty and the commit's content is byte-identical to
the working tree that was audited, so every finding below applies to the commit as-is.

## Method

1. **Diff + full-file read** — the complete v1.17 diff plus the current state of all four
   touched code files, and the relevant BUILD-PLAN.md sections (v1.17 changelog, §4 item 4c).
2. **Independent contrast recomputation** — relative luminance / contrast ratio computed from
   scratch in Node (WCAG 2.x formula) for `--ink` and `--accent` text over the pill's real
   composited background (`--edge` at alpha 0.16 over `--surface` #FFFFFF), both themes,
   both with and without integer rounding of the composite. Not copied from the changelog.
3. **Independent visual verification** — repo served locally (`python3 -m http.server 8792`),
   driven with the pre-installed headless Chromium via Playwright at a real 390×844 mobile
   viewport (deviceScaleFactor 2, isMobile, hasTouch). `page.clock.install` froze the browser's
   Date at five instants (all 10:00 HKT, outside the pre-09:00 focus window): 31 days out in
   calm and blossom, 1 day out, the trip's own HKT day, and the day after. Full-page and
   Kenya-card-only PNGs were captured and actually inspected, DOM geometry and computed styles
   measured in-page, console errors collected. A sixth, real-current-time pass (09:45 HKT,
   2026-07-15) confirmed the live default state. An `ariaSnapshot()` of the Kenya card was
   taken to check how the pill is actually exposed to assistive tech, not just what attributes
   it carries.
4. **Verifier ratchet check, both sides** — `node scripts/verify.mjs all` run on the feature
   code (62/62) *and*, via a temporary git worktree, at the pre-feature merge commit `bcf0399`
   (61/61), proving the claimed 61→62 is real and that the diff only adds a check (confirmed
   by reading the diff: no existing check was touched).
5. **Mechanical checks** — `node --check` on `app.js`/`lib.mjs`/`scripts/verify.mjs`; byte
   budgets measured directly with `wc -c`/`du`; grep sweeps for `max-width` media queries,
   `localStorage` key usage, and PII in the diff.

## Findings

### Confirmed (implementer claims that survived independent verification)

- **Countdown logic is correct and HKT-anchored (invariant 8).** `daysUntilKenyaTrip` diffs a
  fixed epoch-day number for 2026-08-15 against `hktDayNumber(now)`, which routes through the
  existing `Intl.DateTimeFormat` + explicit `timeZone: "Asia/Hong_Kong"` helper — the same
  idiom as `pickIndex`. Verified at the four boundary instants in `verify.mjs` (31 → 30 across
  an HKT midnight; 0 on the trip's own HKT day from its first minute; −1 the day after) and
  live in the browser at frozen clocks. The trip-day and day-after transitions are exact.
- **All five UI states render as claimed.** `31 DAYS` (calm and blossom), `1 DAY`, `TODAY`,
  and — day after departure — the pill is genuinely absent from the DOM
  (`querySelector(".kenya-countdown") === null`, chip row measured back at its bare 11px), not
  merely invisible. The Kenya fact keeps rendering normally in every state. Zero console
  errors in all six passes; no horizontal overflow (`scrollWidth` 390 = viewport).
- **Contrast: the shipped color choice passes AA with huge margin, and the rejected
  alternative really would have failed.** Recomputed from raw tokens: `--ink` on the composited
  pill background = **13.70:1 (calm)** and **13.74:1 (blossom)** — the 10px pill text clears
  the 4.5:1 normal-text bar roughly three times over. `--accent` on the same background =
  **4.46:1 (calm)** and **4.07:1 (blossom)** — both genuinely below 4.5:1, so ruling it out was
  the right call, independently re-derived. (Exact figures vs. the changelog's: see Corrected
  below.) Computed pill styles in-browser match the tokens exactly
  (`rgba(43,95,217,0.16)` / `rgba(201,79,124,0.16)` backgrounds, theme `--ink` text).
- **Verifier ratchet honored (invariant 12).** 61/61 at pre-feature `bcf0399` (run in a temp
  worktree), 62/62 on the feature — one check added, none modified or loosened, so no
  `decisions.md` entry is required. `node --check` clean on all three touched JS files.
- **Budgets all pass (invariant 6), measured directly:** JS total 23,201 B ≤ 60 KB
  (`figure.js` 7,622 B ≤ 12 KB, untouched); fonts 187,267 B ≤ 300 KB; page excl. fonts
  146,221 B ≤ 350 KB. `verify.mjs` stage5 agrees.
- **Remaining invariants: clean.** No PII in the diff (a trip date plus the long-established
  Kenya card theme is outside the enumerated PII scope — no name/employer/DOB/location/
  finances anywhere); no quotation-mark glyphs in any new string; zero runtime deps added;
  `localStorage` untouched; no media query touched (and no `max-width` query exists);
  `noindex` and `.nojekyll` untouched; the pill is correctly non-interactive — no tap-target
  or focus-ring obligation attaches to it, and it doesn't sit inside or obscure any
  interactive element.

### Corrected (claims that did not fully survive; none change the verdict)

- **The changelog's blossom contrast figures are slightly wrong.** Claimed: `--ink` 13.55:1,
  `--accent` 4.01:1. Recomputed: **13.74:1** and **4.07:1** (the calm figures, 13.70:1 and
  4.46:1, reproduce exactly). Neither plausible alternate basis (compositing over `--bg`
  instead of `--surface` gives 12.79/3.79) reproduces the claimed numbers, so they appear to
  be simple computation slips. Immaterial to every decision — `--accent` fails AA and `--ink`
  passes enormously under both the claimed and actual figures — but BUILD-PLAN.md is the
  project's source of truth and should carry the right numbers.
- **"The pill adds no height to the card's chip row" is false as written.** Measured: the bare
  `.card-chip` row is 11px tall; with the pill present, `.card-top` is 19px — the Kenya card
  gets ~8px taller while the countdown is showing (the pill's 10px text + 3px vertical padding
  exceeds the 9px chip). Visually this is harmless — screenshots show the row reads as one
  balanced line, `align-items: center` keeps the chip optically placed, and nothing shifts in
  the other cards — but the changelog's claim is inaccurate and should be corrected rather
  than left to mislead a future audit.

### Minor, non-blocking observations

- **`aria-label` on a role-less `<div>` is unreliable naming.** ARIA 1.2 prohibits naming on
  the `generic` role, and the empirical `ariaSnapshot()` of the card bears this out: the pill
  is exposed as plain text — the card reads `KENYA 31 DAYS` — with the label not surfacing in
  the tree. Practical harm is near zero (`KENYA 31 DAYS` adjacent to the fact is perfectly
  comprehensible, arguably closer to the sighted experience), so this does not block. If the
  full sentence is wanted for screen readers, the robust pattern is a visually-hidden
  `<span>` alongside an `aria-hidden` visible label, or an element whose role permits naming.
- **The countdown is computed once per load**, like every other time-gated render here (the
  BUILD-PLAN states this convention explicitly for v1.16). The `visibilitychange` resume
  handler only repaints when the pre-09:00 focus boundary flips, so an iOS PWA resumed the
  next day within the same focus state shows yesterday's count until reload — but it equally
  shows yesterday's cards and date line, a pre-existing, documented trade-off this feature
  merely inherits. If it ever bothers anyone, the same handler could also compare
  `hktDayNumber`; not needed now.
- **First real use of the IBM Plex Mono 500 face.** The `@font-face` and the 14.9 KB woff2
  were already shipped and already counted in the font budget, but nothing used weight 500
  until this pill, so browsers will now actually download it. Within budget; informational
  only.

### Design judgment (the "beautiful UI" question, not just mechanics)

The pill looks intentional, not bolted on. It speaks the app's existing dialect — rounded
mono-text pills are already the vocabulary of the theme toggle, the staleness chip, and the
reveal-rest disclosure — and the `--edge` tint keys it to each theme's accent (soft blue in
calm, soft rose in blossom) without shouting. At 390×844 it reads instantly as "badge on the
Kenya card," sits on the same optical line as the KENYA chip, and never competes with the
Fraunces fact body, which stays the card's clear focal point. Top-right is the right
placement: it pairs the ambient metadata (what card / how long to go) on one quiet line and
leaves the reading flow untouched; putting it near the attribution line or below the fact
would bury time-sensitive information, and a full-width banner would overweight it. The
asymmetry against Anchor/Journal/Word is fine — it is motivated asymmetry, exactly like the
Word card being the only one with a headline and a speak button; only Kenya has a real event
date attached. The `TODAY` state is a nice touch of restraint (no exclamation, same pill), and
disappearing entirely after the trip — verified absent, not blank — is the correct end state.

## Verdict

**Ship with one minor follow-up.** All functional, accessibility, invariant, and budget checks
pass under independent re-verification; the design succeeds on its own terms and in context.
The single follow-up is textual, not code: correct the two inaccurate sentences in
BUILD-PLAN.md's v1.17 changelog (the blossom contrast figures — 13.74:1 / 4.07:1, not
13.55:1 / 4.01:1 — and the "adds no height to the chip row" claim, which should say the row
grows ~8px while the pill is visible). BUILD-PLAN.md is the declared source of truth; it
should not carry numbers a future auditor will fail to reproduce. Nothing else blocks the
merge.
